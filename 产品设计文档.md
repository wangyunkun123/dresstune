## 一、产品定位

### 1.1 一句话

**每天 1 个穿搭微调。不教你买什么，教你「怎么穿」。**

### 1.2 核心主张

市面上的穿搭产品都在解决一个问题：「你应该穿什么衣服」。我们解决另一个问题：「**你已经有衣服了，怎么把它穿好看**」。

同样的白衬衫 + 牛仔裤——塞不塞、卷不卷、解几颗扣——是路人还是博主的差距。这些微小的穿搭技巧，才是普通用户的时尚盲区。我们在做的就是：**时尚最后一公里**。

### 1.3 目标用户

**核心用户**（产品主要服务对象）：
- 已经会基本穿搭，但感觉"差一口气"的人
- 照着博主买同款，穿上就是不对味的人
- 想提升穿搭水平，但没有精力和预算系统学习的人

**延伸用户**（产品可以服务但不重点优化）：
- 每天早上站在衣柜前纠结"今天穿什么"的人（Layer 2 解锁后自然覆盖）
- 完全不懂穿搭的新手（纠正模式兜底，但不针对此人群优化）

> 产品设计和资源分配聚焦核心用户。纠正模式（🔴）作为兜底逻辑存在，不宣传、不做专项优化。当用户水平从新手成长到"会基本穿搭"时，自然进入核心用户体验区。

### 1.4 使用门槛

- ❌ 不需要录入衣橱
- ❌ 不需要填写问卷
- ❌ 不需要关注博主
- ✅ 首次：拍 3 张身形照（60 秒）
- ✅ 每天：拍 1 张穿搭照（15 秒）

### 1.5 为什么通用 AI 做不到

这是我们的护城河。ChatGPT、豆包、Claude 这些通用 AI 对话产品，每一个单独能力都很强，但它们做不到：

| | 我们 | 通用 AI 对话 |
|---|------|------------|
| 持续追踪你的穿搭演进 | ✅ 每张照片都在积累 | ❌ 每次对话从零开始 |
| 在你真实身形上生成效果图 | ✅ BEFORE/AFTER 在你身上 | ❌ 文字描述或通用模特 |
| 结构化技巧课程 | ✅ 100+ 技巧 × 分级 × 解锁 | ❌ 碎片化、无进度 |
| 被动认识你的衣橱 | ✅ 穿搭快照自动积累 | ❌ 不知道你有什么 |
| 知道你掌握了什么 | ✅ 技巧树 × 掌握清单 | ❌ 只有当前对话 |
| 让你觉得自己在进步 | ✅ 打卡/连胜/成就/进化时间线 | ❌ 没有习惯养成引擎 |
| 技巧可携带离开 App | ✅ 学到的本事是自己的 | ❌ 看完就忘 |

**核心差异一句话**：通用 AI 能回答「今天穿什么」，我们能让用户「每天变好一点点，并且看得见」。

## 二、核心产品逻辑

### 2.1 三层递进架构

```
┌─────────────────────────────────────────────┐
│                                             │
│  Layer 1：学会穿                              │
│  拍照 → AI审稿 → 微调建议 → 前后对比 → 打卡     │
│                                             │
│  用户获得：可以带走的穿搭技巧 + 肉眼可见的进步    │
│  产品获得：身形数据 · 衣橱数据 · 风格偏好 · 行为数据│
│  ⏱️ Day 1 ~ Day 30                          │
│                                             │
│                  ↓ 数据触达，AI主动邀请       │
│                                             │
│  Layer 2：帮你穿                              │
│  AI比你更了解你的衣橱和风格 → 每日3套方案         │
│  左滑右滑 → 数据回流 → 越来越准                 │
│  ⏱️ Day 30+                                 │
│                                             │
│                  ↓ 衣橱数字化完成 + 风格DNA形成 │
│                                             │
│  Layer 3：聪明买                              │
│  AI发现衣橱缺口 →「你缺这件来完成你的风格」       │
│  精准单品推荐，基于真实需求而非猜你喜欢           │
│  ⏱️ Day 60+                                 │
│                                             │
└─────────────────────────────────────────────┘
```

### 2.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **渐进式数据采集** | 不要求用户一次性录入，数据在日常使用中自然积累 |
| **先给价值再取数据** | 每个交互都给用户即时可见的价值，然后默默记录数据 |
| **穿搭快照代替衣橱** | 不提取单品，AI 输出结构化穿搭描述（品类+颜色+版型），衣橱从"单品库"变成"穿搭日历" |
| **AI 是审稿人，不是裁判** | AI 不否定用户的穿搭，在已有基础上优化。即使需要纠正，也只说"这条裤子不太适合你"而非"你穿得不对"。评分数字和颜色标记对用户隐藏 |
| **技巧可携带** | 用户学到的是自己的本事，离开 App 也能用 |

**第五步的设计意图**：让用户感知到「技巧可携带」——今天不只是拍了一张照，是学会了一个能带走的技能。这是产品最深层的情感价值：不是「AI 帮你变好看」，是「你变厉害了」。

### 3.2 AI 审稿引擎

**架构原则**：3 段式独立管线，每段独立运行、独立降级。不串联依赖——段一失败不影响段二，核心渲染层失败不影响段一二。

**重要定位**：Checklist 分流评估的是「今天这套穿搭的状态」，不是「用户的能力等级」。同一个用户，周一可能 🔴 纠正，周五可能 🟢 精进。用户不需要选择等级——系统每天自动判断。

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  段一：穿搭描述（高可靠度 ~90%）                        │
│  ┌────────────────────────────────────────────────┐  │
│  │ 输入：用户照片                                    │  │
│  │ 处理：VLM 一次调用 → 品类+颜色+版型+穿着方式+ Checklist│  │
│  │ 输出：穿搭描述 JSON + 7 项 Yes/No 评估             │  │
│  │ 用户可见：✅「AI 识别到：白T恤 + 深蓝牛仔裤 + 帆布鞋」 │  │
│  │ 失败处理：用户手动修正品类标签（下拉选择器，3秒）      │  │
│  └────────────────────────────────────────────────┘  │
│                         │                            │
│                         ▼                            │
│  段二：建议生成（中可靠度 ~80%）                        │
│  ┌────────────────────────────────────────────────┐  │
│  │ 输入：穿搭描述 JSON + 身形参数 + 技巧历史            │  │
│  │ 处理：Checklist 分流 → LLM 生成建议文本             │  │
│  │ 输出：微调建议文字 + 静态分步图解                    │  │
│  │ 用户可见：✅「试试法式半塞」+ 3 张步骤图             │  │
│  │ 失败处理：降级为品类匹配的通用技巧推荐                │  │
│  └────────────────────────────────────────────────┘  │
│                         │                            │
│                         ▼                            │
│  核心渲染层：效果图生成（Phase 1 重点投入，目标 95% 出图率）  │
│  ┌────────────────────────────────────────────────┐  │
│  │ 输入：用户原图 + 微调描述                         │  │
│  │ 处理：难度分级 → L1 inpainting / L2 叠加 / L3 重生成│  │
│  │ 输出：效果图（或降级为静态参考图）                   │  │
│  │ 用户可见：✅「AI 模拟效果」+ 「AI 模拟，主要看结构变化」│  │
│  │ 失败处理：展示静态参考图 + 文字指导，不出效果图       │  │
│  └────────────────────────────────────────────────┘  │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**关键设计决策**：
- 为什么 3 段而不是 8 步？8 步串联可靠性 0.9⁸=43%，3 段每段独立降级后，用户不会看到「整个流程崩了」
- 为什么 Checklist 而不是连续打分？VLM 对「是/否」的判断准确度远高于「打几分」。Yes/No 分类边界明确，不同调用之间一致

**段二 Checklist（7 项分流评估）**：

```json
[
  {"id":"fit_top",     "question":"上装合身吗？",       "result":"YES","reason":""},
  {"id":"fit_bottom",  "question":"下装合身吗？",       "result":"NO", "reason":"裤长过长堆叠"},
  {"id":"fit_shoe",    "question":"鞋与整体比例协调吗？","result":"YES","reason":""},
  {"id":"color_match", "question":"颜色搭配协调吗？",    "result":"YES","reason":""},
  {"id":"style_match", "question":"上下装风格一致吗？",  "result":"YES","reason":""},
  {"id":"prop_balance","question":"上下身比例舒服吗？",  "result":"NO", "reason":"腰线被遮住显腿短"},
  {"id":"occasion",    "question":"适合日常出门吗？",    "result":"YES","reason":""}
]
```

**分流规则**（基于 ❌ 数量，一次 VLM 调用同时输出）：

```
0 个 ❌ → 🟢 精进模式：推荐进阶技巧
1-2 个 ❌ → 🟡 微调模式：推荐与 ❌ 项相关的微调技巧
3+ 个 ❌ → 🔴 纠正模式：优先解决最严重的那个 ❌ 项
```

**评分映射**（后台使用，不对用户展示）：

```
穿搭评分 = (7 项中 YES 的数量 / 7) × 100

示例：5/7 YES → 71 分，3/7 YES → 42 分

此分值只用于：
  - 穿搭进化曲线（个人主页/周报的趋势图）
  - 后台分流逻辑（不给用户看数字）
  
用户看到的是自然语言建议，不是打分。
```

**个性化防趋同** 🆕：

行业研究表明 AI 穿搭推荐会不自觉推动用户走向同质化（"所有人的建议都是浅色西装+直筒牛仔裤+白球鞋"）。我们的应对：

```
技巧推荐排序 = 效果显著度 × 执行容易度 × 水平匹配度 × 个性化因子

个性化因子计算：
  基础权重 1.0
  - 该技巧在全体用户中被推荐次数越多 → 权重略微降低（-0.1 ~ -0.3）
  - 该技巧与用户风格方向契合度越高 → 权重提升（+0.1 ~ +0.3）
  - 该技巧在用户近期（7天内）已被推荐过 → 权重降低（-0.5，避免重复）
  - 用户对该类技巧的历史反馈"不如"比率高 → 权重降低（-0.3）

结果：同一套白T恤+牛仔裤，
  Clean Fit 用户 → 可能推荐「法式半塞」
  City Boy 用户 → 可能推荐「宽松堆叠不塞」
  韩系轻熟用户 → 可能推荐「规整全塞 + 卷边」
```

同一套衣服，不同风格方向给不同的微调——这是技巧库与风格知识库联动的核心价值。

**段三效果图分级生成策略**：

| 难度 | 调整类型 | 示例 | 生成策略 | 保真度 |
|------|---------|------|---------|--------|
| L1 | 穿着方式调整 | 塞衣、卷裤脚、解扣子 | 原图 + inpainting 局部重绘 | ⭐⭐⭐⭐ |
| L2 | 配饰添加 | 加腰带、换鞋 | 原图 + 配件叠加合成 | ⭐⭐⭐ |
| L3 | 单品替换 | 换一条裤子 | Seedream 全身重生成 → 过质检 | ⭐⭐ |
| L4 | 全身改造 | 完全重构 | 不生成效果图，只给文字+参考图 | ⭐ |

**L3 质检门槛**：体型相似度 > 80% + 肤色偏差 < 阈值 + 背景一致。不通过则降级为静态指南。

**降级展示**：效果图生成失败时，展示「静态分步图解 + 文字指导」，标注「这个调整不适合 AI 模拟，以下是操作指南」。

**离线/弱网处理** 🆕：

| 场景 | 策略 |
|------|------|
| 拍照 | 始终可用。照片本地缓存，网络恢复后自动上传分析 |
| AI 分析 | 需要网络。弱网时排队等待，不阻塞拍照 |
| 效果图生成 | 需要网络。生成中展示进度动画，超时 15s 自动降级为静态指南 |
| 打卡 + 反馈 | 本地记录，网络恢复后同步。打卡不丢 |
| 浏览历史 | 近期快照本地缓存，离线可浏览过去 7 天 |

用户拍照时不应因为网络问题而受阻。核心体验链路「拍照 → 看到建议」中，拍照本身是零网络依赖的。

### 3.3 技巧知识库结构

```
技巧库 🌳

🌱 基础篇：衣服怎么穿（门槛：第1天起）
├─ 塞法系列
│   ├─ 全塞（Tuck All）：全塞进下装，拉出少许松量
│   ├─ 法式半塞（French Tuck）：只塞前摆，后摆自然垂落
│   └─ 随意前塞：只塞正前方一小块，其余自然堆叠
├─ 裤脚系列
│   ├─ 单折卷边：向外翻折一次，宽约2-3cm，整洁感
│   ├─ 双折细卷：Pinroll，向内折细，锥形裤专用
│   └─ 随意堆叠：不卷，自然堆叠在鞋面上，街头感
├─ 袖口系列
│   ├─ 基础翻折：解开袖口，外翻两圈，露出手腕
│   ├─ 推肘堆叠：长袖推到肘部，制造随意层次
│   └─ 卷袖至肘：规整地卷至肘下，工装风格
├─ 领口系列
│   ├─ 扣子法则：衬衫最上两颗不扣，Polo最上一颗不扣
│   ├─ 立领造型：外套领立起，防风+造型
│   └─ 叠穿领口：内层领口高出外层的分寸
└─ 腰带系列
    ├─ 颜色呼应：腰带颜色与鞋/包/表的呼应规则
    ├─ 随意垂坠：不穿过扣环，腰带随意垂下
    └─ 无腰带造型：什么时候不需要系腰带

🌿 进阶篇：比例怎么调（门槛：掌握3个基础技巧）
├─ 上短下长
│   ├─ 视觉高腰：通过塞衣/短款/高腰裤拉高腰线
│   └─ 内搭短外搭长：叠穿中的比例控制
├─ 层次叠穿
│   ├─ 三露法则：领口露·下摆露·袖口露
│   └─ 长度递进：内→中→外 长度依次递增
├─ 露肤分寸
│   ├─ 露踝法则：九分裤/卷边，露出最细的脚踝
│   ├─ 露腕法则：袖口处理，露出手腕
│   └─ 不露的时机：正式场合/寒冷天气的替代方案
└─ 松紧平衡
    ├─ 上宽下窄：宽松上装+修身下装
    ├─ 上窄下宽：修身/塞衣上装+宽松下装
    └─ 全宽收腰：全宽松+腰带收腰

🌳 高级篇：风格怎么塑（门槛：掌握8个技巧+Layer 2解锁）
├─ 色彩桥接
│   └─ 用一件中间色单品，连接上下身对比色
├─ 材质对冲
│   └─ 硬+软·光+糙·厚+薄的对冲搭配法则
├─ 风格混搭
│   └─ 正式+休闲·复古+现代·运动+优雅
├─ 配饰点睛
│   └─ 项链/手表/戒指/包/帽子/袜子的位置与呼应
└─ 造型破格
    └─ 故意不塞·故意不对称·故意堆叠——知道规则后打破它

🎯 大师篇：自己怎么创（门槛：掌握15个技巧+90天+，数据自动触发）
├─ 穿搭签名：建立你独有的穿搭印记
├─ 色彩人格：找到你的专属色系组合
├─ 趋势翻译：把秀场趋势变成你的日常版本
└─ 规则即兴：在了解规则的基础上自由创作

> 大师篇不需要用户手动选择或升级。系统检测到用户满足条件（15 个技巧 + 90 天）后，AI 建议自然切换到大师级语言——不再是「试试法式半塞」，而是「你的穿搭签名是 Clean Fit + 日系层次，这周可以尝试在配饰上做实验」。
```

**打卡反馈：三元微反馈** 🆕

用户拍完调整后的第二张照片时，弹出极简反馈：

```
┌──────────────────────────────────────┐
│  ✅ 打卡完成！Day 15                  │
│                                      │
│  ┌──────────┐    ┌──────────┐       │

│                                      │

│                                      │
└──────────────────────────────────────┘
```

**为什么是三元而不是评分**：
- 一次点击，零负担
- 「更好/差不多/不如」是比接受率更强的训练信号——它直接衡量建议的有效性
- 如果某个技巧的「不如」反馈率 >30%，AI 降低该技巧对当前用户的推荐优先级
- 全局「不如」率异常高的技巧 → 触发教学内容优化

**反馈数据使用**：
```json
{
  "skill_id": "tuck_french",
  "feedback": "better",  // better | same | worse | null
  "used_in_retraining": true
}
```

- 用户级：某类技巧「不如」率高 → 换同级别其他类型
- 技巧级：全局高「不如」→ 优化该技巧的教学内容
- 时间级：反馈趋势反映用户穿搭水平真实提升

### 4.3 设计要点

- **每天 3 套，不是无限滑动**。AI 的价值在于帮你决策，而不是给你无限选项让你继续纠结
- **第 3 套永远是"舒适区边缘"**——保持探索性，防止用户陷入穿搭惯性
- **右滑 = 采纳** → 明早推送提醒 → 第二天晚上弹窗"今天穿了吗？"→ 记录下来
- **左滑 = 不采纳** → 同风格换另一套（不是随机换，是了解你为什么不喜欢）
- **通过右滑左手滑**的过程继续学习用户偏好，推荐越来越准

 ✅ 帆布鞋 1双

### 6.2 隐私与数据安全

**核心原则**：穿搭照片是用户最敏感的数据资产——比衣橱清单敏感得多。零数据丢失、零滥用、用户完全控制。

**数据分类与存储策略**：

| 数据类型 | 敏感度 | 存储位置 | 加密 | 用户控制 |
|---------|--------|---------|------|---------|
| 身形照片（3张） | 🔴 极高 | 设备端 + 端到端加密云端备份 | AES-256 | 随时删除，删除后身形参数保留 |
| 每日穿搭照片 | 🔴 极高 | 端到端加密云端存储 | AES-256 | 随时删除单张或批量删除 |
| 身形参数（数值） | 🟡 中 | 云端 | 传输加密 | 可导出、可删除 |
| 穿搭快照（JSON） | 🟡 中 | 云端 | 传输加密 | 可导出、可删除 |
| 行为数据 | 🟢 低 | 云端脱敏 | 传输加密 | 可拒绝收集 |

**关键安全措施**：

- **照片不用于模型训练**：用户照片仅用于该用户的 AI 分析和效果图生成，不会成为训练数据。此承诺写入隐私政策
- **身形参数与照片分离**：身形建模完成后，参数以数值形式独立存储。即使照片被删除，身形参数仍可正常工作
- **数据导出权**：用户可随时导出全部穿搭快照 JSON + 身形参数 + 技巧掌握清单，格式为可读的 ZIP 包
- **数据删除权**：用户可删除任意单张照片、全部照片、或整个账号。删除后 30 天内可恢复，30 天后永久清除
- **数据备份**：每日自动备份，多地冗余。穿搭快照是时间积累的心血，绝不可丢失

**行业教训**：
- Combyne 更新删用户数据 → 毁灭性差评 → 用户不会回来
- 灰产售卖身体数据 0.5 元/条 → 身形数据加密等级必须等同金融数据
- AI 试衣"美颜"扭曲自我认知 → 效果图必须保持体型真实性

**效果图保真度硬约束** 🆕：

```
生成效果图时，以下参数不得改变：
  ❌ 体型/身材比例
  ❌ 肤色
  ❌ 面部特征
  ❌ 背景环境
  
允许改变的：
  ✅ 衣服的穿着方式（塞入/卷起/敞开）
  ✅ 配饰的添加/移除
  ✅ 单品的替换（L3 级别，需质检通过）
```

此约束写入 Seedream prompt 模板的 negative prompt，作为不可协商的硬规则。

### 6.3 穿搭快照季节系统

**问题**：纯时间衰减无法处理季节更替。去年夏天的白 T 恤，今年夏天应该重新被推荐，而不是因为过了一年就衰减到零。

**方案**：每张穿搭快照在拍照时自动标记季节和温度标签。

**快照数据结构**：

```json
{
  "date": "2026-07-15",
  "season": "summer",
  "temp_range": "25-35°C",
  "weather": "sunny",
  "items": [
    {"category": "TS", "color": "白色", "fit": "宽松", "note": "圆领短袖"}
  ],
  "outfit_description": "白圆领T恤 + 深蓝直筒牛仔裤 + 白色帆布鞋"
}
```

季节和温度通过天气 API 在拍照时自动写入，用户无感知。

**季节-月份映射**（比四季更细）：

| 月份 | 季节标签 | 推荐偏好 |
|------|---------|---------|
| 12-2月 | winter | 厚外套、毛衣、叠穿技巧 |
| 3-4月 | spring | 薄外套、层次叠穿 |
| 5-6月 | early_summer | 短袖、轻薄面料、露踝技巧 |
| 7-8月 | peak_summer | 最轻薄、露肤技巧 |
| 9-10月 | autumn | 薄外套回归、围巾技巧 |
| 11月 | late_autumn | 厚外套、层叠 |

**检索权重公式**：

```
最终权重 = 基础时间衰减 + 季节修正

基础时间衰减：
  0-14天   → 1.0
  15-30天  → 0.7
  31-60天  → 0.4
  60天以上 → 0.15

季节修正：
  同季节     → +0.5  （去年的夏天快照 → 今年夏天 0.65，正常推荐）
  相邻季节   →  0    （春→夏，自然过渡）
  反季节     → -0.3  （冬→夏，最低 0，不推荐）
```

**效果示例**（假设今天 2027-07-15 夏天）：

| 快照日期 | 季节 | 天数 | 基础 | 修正 | 最终 | 说明 |
|---------|------|------|------|------|------|------|
| 2027-07-10 | 夏 | 5天 | 1.0 | +0 | 1.00 | 最近拍的 |
| 2027-06-20 | 夏 | 25天 | 0.7 | +0 | 0.70 | 上个月 |
| 2027-01-15 | 冬 | 181天 | 0.15 | -0.3 | 0.00 | 反季节，不推荐 |
| 2026-08-10 | 夏 | 339天 | 0.15 | +0.5 | 0.65 | ✅ 去年夏天，提权 |
| 2026-07-15 | 夏 | 365天 | 0.15 | +0.5 | 0.65 | ✅ 去年同一天 |

换季月份（3月/9月），相邻季节修正为 0，两季快照权重接近，自然过渡。

### 6.4 性别内容策略

**核心原则**：架构层面从 Day 1 支持多性别，内容层面 Phase 1 聚焦男性。

**为什么架构要预留**：每条内容加一个 `target_gender` 字段的成本不到半天。但如果现在不加，未来每条内容都要逐条回溯、改 prompt、改数据结构——成本翻 5 倍以上。

**为什么内容先做男性**：
- 原项目已有 49 个男装风格 + 18 个男性评分指纹，内容资产最厚
- 身形分析 + 效果图生成先调好一套，再扩展到女性
- 种子用户聚焦一个群体，跑通核心循环

**技巧库性别分布**：

```
技巧库 ~100 个技巧：

🌍 通用技巧（~40%，男女共用）
   ├─ 塞法系列、颜色搭配规则、比例调节、配饰点睛
   
👔 男性专属（~30%）
   ├─ Polo/衬衫扣子法则、Pinroll 裤脚、西装技巧
   
👗 女性专属（~30%）
   ├─ 裙长比例、腰带腰线、叠穿女性廓形
```

**分阶段计划**：

| 阶段 | 性别 | 内容 |
|------|------|------|
| Phase 1 | 男性 | 15 个基础技巧（通用 60% + 男性专属 40%） |
| Phase 2 | 男性 + 女性 | 女性身形分析 + 50 个女性技巧 + 通用技巧扩展 |
| Phase 3+ | 不限 | 不做儿童（品类体系/身形模型/购买决策完全不同，应独立评估） |

**季节/城市/工作场景策略**：

| 维度 | 处理方式 | 何时做 |
|------|---------|--------|
| 季节 | 拍照时天气 API 自动标记（见 6.3），不影响用户选择 | Phase 1 ✅ |
| 城市 | = 天气数据 + 温度区间，天气 API 自动获取位置。不需要用户手动选择城市 | Phase 1 ✅ |
| 工作场景 | = 风格推荐的过滤条件。可选标签（上班/休闲/运动/约会），非必填 | Phase 2 |

**城市为什么不需要单独做**：用户的穿搭需求由温度 + 天气决定（28°C 晴天 = 轻薄透气，8°C 雨天 = 防水保暖），城市只是获取天气数据的坐标。城市穿搭文化差异（东京 vs 上海）属于风格偏好范畴，由用户选择的风格方向自然覆盖。

## 七、变现模型

### ⚠️ 核心原则：先给价值再取钱

**行业教训**：GetWardrobe、Outfity、Acloset 均因"先让用户投入再索要付费"被骂"骗子"，一星差评无法挽回。用户花了 2 小时录入衣橱后发现核心功能要钱——这种体验是品类公敌。

**我们的原则**：
- 免费层必须是**完整可用的产品**，不是"试用版"
- 付费是对价值的**扩展**，不是对核心功能的**解锁**
- 收费点必须在用户**已经获得价值之后**，不能在此之前
- 所有收费限制在注册时**明确告知**，不搞隐藏付费墙

### 7.1 免费层（永远免费，完整可用）

- ✅ 每日 1 个微调建议 + BEFORE/AFTER 效果图
- ✅ 基础技巧库全部（15 个技巧 + 静态分步图解）
- ✅ 打卡系统 + 三元反馈
- ✅ 个人主页 + 基础成就
- ✅ 穿搭历史回顾（过去 30 天）
- ✅ 每周穿搭报告
- ✅ 身形分析 + 渐进校准

> 免费层本身就是完整产品。用户可以永远免费用下去，每天获得真实价值。

### 7.2 会员层（月付 ¥15 / 年付 ¥128）

**什么时候提示升级**：用户连续打卡 7 天后 → "你已经坚持了一周！要不要解锁更深入的内容？" → 展示会员权益对比。

- 🔓 每日微调 **可追问**（AI 对话式深入解释"为什么这样调整"）
- 🔓 完整技巧库（基础+进阶+高级，~100 个技巧）
- 🔓 进阶技巧的静态分步图解
- 🔓 Layer 2 每日 3 套穿搭推荐（解锁后）
- 🔓 **AI 穿搭私教**：对话式穿搭问答
- 🔓 **30 天风格进化课程**：结构化学习计划
- 🔓 月报深度分析（风格进化曲线 + 单品利用率）
- 🔓 无水印效果图保存
- 🔓 高级成就系统
- 🔓 无广告

**关键设计**：会员的价值在于"更深"和"更多"，不是"能用"。免费用户每天的 1 个微调建议质量与会员完全一样。

### 7.3 品牌合作（B2B，Layer 3 解锁后）

- 脱敏聚合趋势报告：用户群体在学什么技巧/缺什么品类
- 品牌精准推荐位：基于真实衣橱缺口的推荐（非广告位）
- 用户隐私数据永不售卖——B2B 产品基于脱敏聚合统计

### 7.4 MVP 阶段变现策略

Phase 1 MVP **不做付费**。全部功能免费。目标：
- 验证"拍照→微调→打卡"的核心循环是否有用户价值
- 积累真实用户数据（微调接受率、三元反馈、留存曲线）
- 确定哪些功能用户最愿意付费（再做会员设计）

Phase 2 积累足够用户数据后，再上线会员体系。
