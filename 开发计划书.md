## 零、项目命名

### 推荐名称：**搭调** / DressTune

**「搭调」是一个真实的中文词，意思是"和谐、匹配、对路"。**

拆开看：
- **搭** = 穿搭。你的衣品，你的搭配功力。
- **调** = 微调。每天调一点，从路人到博主的那一口气。

合起来：**让你的穿搭更搭调。**

这个词自带产品理念——不是教你买新衣服，是教你把手头已有的衣服穿得更和谐、更对路。而且它不 tech、不浮夸，像一个老朋友跟你说"今天这一身挺搭调的"。

英文 **DressTune**：Dress（穿搭）+ Tune（微调），像给乐器调音一样给穿搭调一调。每天一个 Tune，穿搭自然 Harmony。

### 备选方案

| 名称 | 含义 | 取舍原因 |
|------|------|---------|
| 对镜 | 面对镜子，每日自省 | 诗意但不够直白，首次听到不能联想到穿搭 |
| 一调 | 每日一调 | 太短太抽象，搜索不友好 |
| 穿感 | 穿搭的语感/手感 | 新造词，需要教育用户 |

## 一、产品介绍

### 一句话版（App Store 副标题）

**每天 1 个穿搭微调。不教你买什么，教你「怎么穿」。**

### 简介版（社交媒体 / 推荐语）

> 你有没有过这种经历——跟博主买了一样的白 T 恤和牛仔裤，穿上就是不对味？
>
> 问题不在衣服，在**穿法**。
>
> **搭调**是一个 AI 穿搭微调教练。每天你拍一张今天的穿搭，AI 告诉你一个具体的小调整——塞一下衣角、卷一圈裤脚、解一颗扣子——然后拍一张调整后的照片，看到自己肉眼可见的变化。
>
> 同样的衣服，不一样的穿法。这就是时尚最后一公里。
>
> 🪞 不需要录入衣橱
> 🪞 不需要懂穿搭理论
> 🪞 每天 15 秒，拍一张照就够了
> 🪞 学到的技巧是你的，离开 App 也能用

### 推荐转发版

> 最近在用一个挺有意思的东西叫「搭调」。跟那些让你买买买的穿搭 App 完全相反——它是教你把衣柜里已有的衣服穿好看。
>
> 原理很简单：每天拍一张今天的穿搭 → AI 告诉你怎么微调（比如「把 T 恤前摆塞进裤腰 5cm」）→ 你试一下再拍一张 → 前后对比打卡。
>
> 我用了一周，发现同一件白 T 恤，塞不塞、卷不卷、解几颗扣，差别真的很大。
>
> 适合那种「买了不少衣服但总觉得差口气」的人。

```

教你变潮人                  帮你做自己

专业术语                    大白话

### Phase 0：基础搭建（2-3 天）

**目标**：所有模型调通，照片→解析→建议这条链路跑通。

**状态**：🔴 待启动

#### 0.1 AI Router 层搭建（1 天）

**产出**：`core/ai_router.py`

```
功能：
  - 统一接口调用所有模型
  - 支持 DeepSeek V4 / 豆包 1.8 / MiniMax-M3 / GLM-5.2 / Seedream
  - 自动处理 API 错误和重试
  - 返回统一格式的结构化结果
  - 配置热切换（改 config/models.json 即可换模型）

输入：model_name, system_prompt, user_message, images(optional)
输出：{"success": true, "data": {...}, "model": "deepseek-v4", "latency_ms": 1234}
```

**验收**：
- [ ] 所有 5 个模型都能调通
- [ ] 同一个 prompt 能分别从 DeepSeek V4 和豆包拿到结果
- [ ] 配置文件切换模型无需改代码

#### 0.2 VLM 穿搭解析 Prompt（1 天）

**产出**：`prompts/outfit_parse.md` + 3 张测试照的解析结果

```
System Prompt 核心要求：
  1. 识别品类（TS/SHIRT/JEANS 等代码 + 中文名称）
  2. 识别颜色（主色 + 辅色）
  3. 识别版型（宽松/合身/修身）
  4. 识别穿着方式（塞了没/卷了没/扣子解了几颗）
  5. 完成 7 项 Checklist（每项 YES/NO + 理由一句话）

输出格式：结构化 JSON
```

**验收**：
- [ ] 品类识别准确率 > 85%（你肉眼判断）
- [ ] Checklist 判断基本合理（7 项中至少 5 项你说"对"）
- [ ] 穿着方式（塞没塞、卷没卷）能正确识别

#### 0.3 LLM 建议生成 Prompt（半天）

**产出**：`prompts/suggestion.md` + 基于 0.2 结果的建议文本

```
System Prompt 核心要求：
  1. 基于穿搭 JSON + 身形参数 + 技巧历史
  2. 生成具体的、可操作的微调建议
  3. 语言像朋友在聊天，不用专业术语
  4. 解释"为什么这样调整"（关联身形或比例）
  5. 如果穿搭已经很好了，给进阶技巧而非硬造问题
```

**验收**：
- [ ] 生成的建议你知道「具体要做什么动作」
- [ ] 语言自然，不像 AI 模板
- [ ] 至少 3 次测试中有建议让你觉得「确实可以试试」

### Phase 1：核心管线（1 周）

**目标**：搭建完整的三段式管线，产出文字建议 + 静态教学图解。这是产品核心价值的最小可行单元。

**状态**：🔴 待 Phase 0 完成后启动

#### 1.1 技巧库 MVP 内容生产（3 天）

**产出**：5 个核心技巧的完整教学内容

```
5 个 MVP 技巧：
  1. 法式半塞（French Tuck）
  2. 裤脚卷边（Cuff Roll）
  3. 袖口翻折（Sleeve Fold）
  4. 扣子法则（Button Rules）
  5. 颜色呼应（Color Echo）

每个技巧包含：
  - 技巧 ID / 名称 / 难度
  - 3 步操作说明（每步一句话）
  - 适用品类列表
  - 不适用场景
  - 身形适配说明（什么体型加分/什么体型注意）
  - 风格变体（Clean Fit怎么做 / City Boy怎么做）
  - 最常见的错误做法
  - 3 张步骤示意图（Seedream 生成通用示范图）
```

**生产流程**：
```
DeepSeek V4 生成初稿
    ↓
MiniMax-M3 交叉审核（找错误、找矛盾、找不自然处）
    ↓
你穿上试一遍，拍 BEFORE/AFTER
    ↓
合格 → 入库 / 不合格 → 返回修改
```

**验收**：
- [ ] 5 个技巧全部经过你实测认可
- [ ] AI 互审发现了至少 1 个需要修改的问题
- [ ] 步骤图清晰可用（不需要精美，能看懂就行）

#### 1.2 技巧推荐引擎（1 天）

**产出**：`core/suggestion.py` 中的规则匹配逻辑

```
规则引擎：
  输入：穿搭 JSON + 身形参数 + 技巧历史
  逻辑：
    1. Checklist 分流（0❌精进 / 1-2❌微调 / 3+❌纠正）
    2. 按品类+身形筛选适用技巧
    3. 排除 7 天内已推荐过的技巧
    4. 按效果显著度 × 执行容易度 排序
    5. 取 Top 1
  输出：选中的技巧 ID + 推荐理由
```

**验收**：
- [ ] 同一个穿搭 JSON 输入，两次调用返回不同技巧（多样性）
- [ ] 已标记「掌握」的技巧不会被推荐
- [ ] 不适合品类的技巧不会被推荐

#### 1.3 端到端联调（1 天）

**产出**：5 次完整的「拍照→解析→建议→图解」链路

**验收**：
- [ ] 5 次测试中至少 3 次你觉得建议「有用」
- [ ] 管线在 10 秒内返回完整结果
- [ ] 任何一段失败不会导致整条链路崩溃

#### 1.4 效果图初探（1 天）

**产出**：Seedream 4.0 L1 效果图首次测试结果

```
测试内容：
  - 选 2 张效果最好的穿搭照
  - 每张尝试 2-3 种 prompt 变体
  - L1 目标：塞衣 / 卷裤脚
  
记录：
  - 身份保持了吗？（脸/体型/肤色/背景）
  - 穿着方式变了吗？
  - 人眼看着自然吗？
  - 成功率：生成 10 次，几次能用？
```

**验收**：
- [ ] 有明确结论：L1 效果图成功率大概多少
- [ ] 知道哪些类型的微调最容易/最难生成
- [ ] 决定 Phase 2 走分支 A（优化上线）还是分支 B（降级静态图）

### Phase 2：效果图实验（1 周）

**目标**：基于 Phase 1 结论，决定效果图的产品定位。

**状态**：🔴 待 Phase 1 完成后启动

#### 分支 A：效果图成功率 > 60% → 优化上线

```
任务：
  1. Seedream prompt 优化（找最佳参数组合）
  2. 搭建质检网关（体型/肤色/面部/背景 四项自动检查）
  3. 设计降级策略（质检不通过 → 静态图解，用户无感）
  4. 测量优化后成功率，目标 > 70%
```

#### 分支 B：效果图成功率 < 50% → 降级实验功能

```
任务：
  1. 效果图改为「实验室」功能，用户主动开启
  2. 重点打磨静态图解体验
  3. 评估替代方案：
     a. 真人示范照片（从原项目素材库 + AI 生成通用模特图）
     b. 手绘风格步骤图
     c. 纯文字 + emoji 指导
  4. 产品定位调整为「AI 穿搭教练」不依赖效果图
```

**验收**：
- [ ] 明确决策文档：效果图是核心功能还是实验功能
- [ ] 如果是核心：质检通过率 > 70%
- [ ] 如果降级：静态图解替代方案完成

### Phase 3：App 化（2-3 周）

**目标**：把命令行里的 AI 管线变成手机上的 App。

**状态**：🔴 待 Phase 2 完成后启动

#### 3.1 后端 API（1 周）

**产出**：FastAPI 服务，3 个核心接口

```
POST /api/analyze
  入参：照片 + 用户 ID
  处理：VLM 解析 → 建议生成 → 效果图（可选）
  出参：穿搭描述 + 建议 + 图解 + 效果图 URL

POST /api/feedback
  入参：建议 ID + 反馈类型（better/same/worse）
  处理：写入反馈数据库，更新技巧权重

GET  /api/history?user_id=xxx&days=7
  出参：过去 N 天的穿搭快照列表
```

**验收**：
- [ ] 单次 /analyze 延迟 < 10 秒
- [ ] 并发 10 个请求不崩溃
- [ ] 错误处理：任何一段失败返回部分结果 + 降级标识

#### 3.2 前端 MVP（1-2 周）

**平台选择**：
| 选项 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| 微信小程序 | 审核门槛低，分享天然闭环 | 限于微信生态 | ⭐⭐⭐ |
| React Native | 跨平台 iOS+Android | 审核门槛高 | ⭐⭐ |
| PWA Web App | 最快上线 | 体验不如原生 | ⭐ |

**MVP 页面**：
```
1. 首页：今日穿搭拍照入口
2. 分析中：等待动画（3-10 秒）
3. 结果页：穿搭描述 + 微调建议 + 效果图/图解
4. 打卡页：调整后拍照 + 三元反馈 + 分享
5. 个人页：打卡天数 + 掌握技巧 + 穿搭历史
```

**验收**：
- [ ] 手机上完成「拍照→看建议→打卡」完整流程
- [ ] 拍照到看到结果 < 15 秒（含网络+AI处理）
- [ ] 可以发给 3-5 个人试用

画像：男性，25-35 岁，对穿搭有一定意识但觉得差口气
来源：朋友圈 / 小红书 / 即刻 / 穿搭群
激励：免费使用 + 1对1 反馈通道
```

#### 4.2 数据收集

```
核心指标：
  - 首次设置完成率
  - Day 1 打卡率
  - Day 7 留存
  - 微调接受率（三元反馈 better 占比）
  - 平均使用时长
  - 分享率

定性反馈：
  - 建议有没有用？
  - 哪里觉得不爽？
  - 会不会推荐给朋友？为什么/为什么不？
```

#### 4.3 迭代

```
Week 1：收集反馈 + 快速修 bug

```

## 六、成功标准

### 每个 Phase 的 Go/No-Go 标准

| Phase | Go 条件 | No-Go 条件 |
|-------|---------|-----------|
| 0→1 | VLM 品类识别 > 85%，建议文本可操作 | 多次修改 prompt 后仍 < 60%，产品核心假设可能不成立 |
| 1→2 | 5 次测试中 ≥ 3 次你觉得「有用」 | < 2 次觉得有用，需要重新审视技巧库或 prompt 策略 |
| 2→3 | 效果图定位明确（无论分支 A 还是 B） | 决策摇摆不定，需要更多数据 |
| 3→4 | App 核心流程可用，3 人试用无阻断 bug | 体验太差不敢给人用，需要继续打磨 |
| 4→发布 | Day 7 留存 > 30%，微调接受率 > 40% | Day 7 留存 < 15%，核心循环不成立 |

### MVP 上线标准

- [ ] 10 个种子用户完成首次设置
- [ ] Day 1 打卡率 > 60%
- [ ] 平均建议满意度 > 50%
- [ ] 无阻断级 bug
- [ ] 隐私政策 + 服务协议就绪

## 七、风险管理

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| VLM 穿搭解析准确率不足 | 中 | 高 | 用户手动修正品类（3 秒下拉选择器），Checklist 可以覆盖品类误差 |
| 效果图质量不可接受 | 高 | 中 | 产品不依赖效果图，随时可降级为静态图解 |
| Seedream API 不可用 | 低 | 高 | 效果图不是地基，无效果图产品仍可正常工作 |
| DeepSeek V4 Vision 不稳定（beta） | 中 | 中 | MiniMax-M3 和豆包 Vision 作为备选 |
| 单人开发进度慢 | 高 | 中 | GLM-5.2 + Kimi-K2.7-Code 辅助写代码，AI 处理大部分重复工作 |
| 种子用户招募困难 | 中 | 中 | 从朋友圈开始，1 个满意的用户会带来 3 个 |
| 内容质量被用户识破（AI 生成） | 中 | 高 | 每个技巧你实测过，AI 互审过，质量有底线 |
