# 技术可行性评估 v1.0

> 评估日期：2026-06-29
> 评估范围：Fashion 2.0「穿搭微调教练」全部 AI 依赖项
> 核心问题：当前大模型能力能否完美实现产品设计文档 v0.7 中的效果

---

## 一、AI 依赖全景图

产品设计文档中涉及的 AI 能力可分为 5 个独立模块：

```
用户拍照
    │
    ├─ [1] VLM 穿搭解析 ──→ 品类+颜色+版型+穿着方式 + 7项Checklist
    │                       可靠性目标：~90%
    │
    ├─ [2] LLM 建议生成 ──→ 基于穿搭JSON + 身形 + 技巧历史 → 微调建议文本
    │                       可靠性目标：~80%
    │
    ├─ [3] 效果图生成 ────→ 同人同衣不同穿法的 BEFORE/AFTER 效果图
    │                       可靠性目标：95% 出图率 ⚠️
    │
    ├─ [4] 身形分析 ──────→ 3张照片 → 体型分类 + 比例参数 + 肤色
    │                       可靠性目标：渐进校准
    │
    └─ [5] 天气标签 ──────→ 拍照时自动获取 season/temp/weather
                            可靠性目标：99%
```

## 二、逐项可行性评估

### 模块 1：VLM 穿搭解析

**产品需求**：从用户照片中识别品类、颜色、版型、穿着方式，同时完成 7 项 Yes/No Checklist 评估。

**当前技术能力**：

| 模型 | 视觉能力 | 中文理解 | 成本/图 |
|------|---------|---------|---------|
| GPT-4o | ⭐⭐⭐⭐⭐ 顶级 | ⭐⭐⭐⭐ | ~$0.005 (high detail) |
| Claude 4 | ⭐⭐⭐⭐⭐ 顶级 | ⭐⭐⭐⭐ | ~$0.005 |
| 豆包 1.8 Vision | ⭐⭐⭐⭐ 优秀 | ⭐⭐⭐⭐⭐ 母语 | ~¥0.001 |
| Gemini 2.5 Pro | ⭐⭐⭐⭐⭐ 顶级 | ⭐⭐⭐⭐ | ~$0.003 |

**可行性评级**：🟢 **高可行**

- 服装识别是 VLM 的成熟能力。品类、颜色、版型识别准确率可达 90%+
- 7 项 Yes/No Checklist 的设计恰到好处——VLM 对二元判断的准确度远高于连续打分（85-90% vs 60-70%）
- 产品文档已设计失败处理（用户手动修正品类标签），段一失败不影响段二
- 豆包 1.8 Vision 在中文场景性价比最优，输入仅 ¥0.80/M tokens

**风险点**：
- 用户拍照质量参差不齐（光线暗、镜子反光、角度偏）会显著影响准确率
- 需要做图片质量预检（模糊检测、光线检测、全身可见检测）

**结论**：可行。但需要图片质量门控作为前置步骤。

---

### 模块 2：LLM 建议生成

**产品需求**：基于穿搭描述 JSON + 身形参数 + 技巧历史 + 天气 → 生成个性化微调建议文本。

**当前技术能力**：

| 模型 | 文本生成 | 结构化输出 | 成本/次 |
|------|---------|-----------|---------|
| GPT-4o Mini | ⭐⭐⭐⭐ | ✅ JSON Mode | ~$0.001 |
| 豆包 1.8 | ⭐⭐⭐⭐ | ✅ | ~¥0.003 |
| Claude Haiku 4.5 | ⭐⭐⭐⭐ | ✅ | ~$0.001 |
| DeepSeek V3 | ⭐⭐⭐⭐⭐ | ✅ | ~¥0.002 |

**可行性评级**：🟢 **高可行**

- 纯文本生成任务，当前 LLM 完全胜任
- 真正的挑战不在 AI 能力，而在**技巧库内容质量**和**推荐匹配逻辑**
- 个性化防趋同公式是算法问题，不依赖 AI

**风险点**：
- 建议文本的「人味」和多样性需要 prompt engineering 调优
- 如果所有建议听起来像模板，用户会很快流失
- 需要建立技巧-场景-身形-风格的匹配规则库，这是内容工作而非 AI 工作

**结论**：可行。核心挑战是内容工程而非 AI 能力。

### 模块 3：效果图生成 🔴 关键风险

**产品需求**：在同一用户身上展示「调整前 vs 调整后」的效果对比。

硬约束（不可改变）：
- ❌ 体型/身材比例
- ❌ 肤色
- ❌ 面部特征
- ❌ 背景环境

允许改变：
- ✅ 衣服的穿着方式（塞入/卷起/敞开）
- ✅ 配饰的添加/移除
- ✅ 单品的替换（L3 级别）

**L1-L4 难度分级**：

| 难度 | 调整类型 | 示例 | 预期策略 | 技术可行性 |
|------|---------|------|---------|-----------|
| L1 | 穿着方式调整 | 塞衣、卷裤脚、解扣子 | 原图+inpainting | 🟡 中等 |
| L2 | 配饰添加 | 加腰带、换鞋 | 原图+配件叠加 | 🟢 较高 |
| L3 | 单品替换 | 换一条裤子 | 全身重生成+质检 | 🟠 较低 |
| L4 | 全身改造 | 完全重构 | 不出图，文字+参考图 | 🟢 (不生成) |

#### L1 深度分析：穿着方式调整

**这是产品最高频的使用场景（塞衣、卷裤脚、解扣子、翻袖口），也是技术上最独特的挑战。**

为什么独特：
- 传统虚拟试穿（VTON）解决的是「换一件衣服」——输入新衣服图片，生成穿着效果
- 我们需要的是「同一件衣服，不同穿法」——没有「新衣服」作为输入，只有穿着方式的改变
- 这要求模型理解衣物的 3D 几何结构：塞入 = 布料进入裤腰、卷起 = 布料折叠

**学术界最接近的方案**：

| 方案 | 核心思路 | 效果 | 生产就绪度 |
|------|---------|------|-----------|
| **InstructVTON** (Amazon/UCLA, 2025) | VLM解析指令 → 自适应遮罩 → 多轮inpainting | 明确支持 "sleeves rolled up" "tucked in" | 研究阶段 |
| **PromptDresser** (KAIST, ICCV 2025) | 文本可编辑VTON，prompt感知遮罩 | 支持 "fully tucked" "untucked" "sleeves rolled" | 代码开源，需适配 |
| **Seedream 4.0 Edit** (字节, 2025) | 自然语言编辑+身份保持+inpainting | 换装/配饰编辑优秀，身份保持强 | ✅ 生产可用 API |
| **FLUX.1 Kontext** (Black Forest, 2025) | 指令式编辑，无需手动遮罩 | "change shirt to black" 类编辑优秀 | ✅ 生产可用 API |
| **GPT-4o Image Gen** (OpenAI, 2025) | 自然语言多轮编辑 | 编辑能力强，但身份一致性弱 | ✅ API 可用，贵 |

**关键发现**：

1. InstructVTON 论文直接讨论了我们的核心场景：卷袖子需要两步——先生成穿背心的中间图（露出手臂），再在手臂上生成卷起袖子的衬衫。这说明即使是学术界最优方案，也需要多步推理。

2. Seedream 4.0 的身份保持能力最强（多参考图锁定），但「同衣不同穿法」不是它的标准用例。它更擅长「换一件衣服」而非「同一件衣服换穿法」。

3. 所有现有方案的共同局限：**无法保证 95% 的一次生成成功率**。实际测试中，inpainting 类方案在穿着方式调整上的首次成功率约 50-70%。

#### L2 分析：配饰添加

**可行性较高**。添加腰带、换鞋是标准的 inpainting/编辑任务。Seedream 4.0 Edit 和 Flux Kontext 都能较好地处理。成功率预计 70-85%。

#### L3 分析：单品替换

**可行性较低**。换裤子需要全身重生成，身份一致性难以保证。产品文档已设定质检门槛（体型相似度 > 80%），但这个门槛本身就很难过。建议 MVP 阶段不做 L3，降级为「穿类似裤子的参考模特图 + 文字说明」。

#### 效果图综合评估

| 维度 | 评估 |
|------|------|
| 能做出来吗？ | ✅ 能，但**不是每一次都能** |
| 95% 出图率可实现吗？ | ❌ 当前做不到。实际预计 60-75%（L1/L2 混合） |
| 保真度硬约束能满足吗？ | 🟡 身份保持可以（Seedream），但衣物编辑的准确性不稳定 |
| 用户体验可接受吗？ | 🟡 需要管理预期——标注「AI 模拟，主要看结构变化」是对的 |

**可行性评级**：🟡 **有条件可行（产品核心风险）**

### 模块 4：身形分析

**产品需求**：从 3 张照片（正面+侧面+穿搭照）分析体型分类、肩胯比、腰线、腿身比、肤色。

**当前技术能力**：

| 方案 | 精度 | 成本 | 生产就绪 |
|------|------|------|---------|
| MediaPipe Pose + 几何计算 | ±2-3cm | 免费 | ✅ |
| VLM 体型分类 | 78-97% 准确率 | ~¥0.001/次 | ✅ |
| Size Stream API | ±1cm | 商业授权 | ✅ |
| 自建融合方案 | 最高 | 中等 | 需开发 |

**可行性评级**：🟢 **高可行**

- 我们不需要精确到厘米的身体测量
- 我们需要的是：体型分类（倒三角/矩形/梯形等）+ 比例关系（肩胯比、腿身比）+ 肤色类型
- VLM + MediaPipe 融合方案完全够用
- 产品文档已设计置信度渐进提升 + 用户手动修正机制

**结论**：可行。MVP 用 VLM + MediaPipe 方案。

## 四、效果图生成的务实路径

### 方案对比

| 方案 | 思路 | L1 成功率 | 成本/图 | 开发量 | 推荐 |
|------|------|----------|---------|--------|------|
| **A: Seedream 4.0 直出** | inpainting API，遮罩+提示词 | 50-65% | ¥0.2-0.3 | 2周 | ⭐⭐⭐ |
| **B: InstructVTON 自建** | 复现论文，多轮agentic inpainting | 60-75% | GPU成本 | 8-12周 | ⭐⭐ |
| **C: FLUX Kontext** | 指令式编辑 API | 55-70% | $0.01-0.03 | 3周 | ⭐⭐⭐ |
| **D: 混合流水线** | Seedream+FLUX双引擎+质检路由 | 70-80% | ¥0.3-0.5 | 6-8周 | ⭐⭐⭐⭐ |
| **E: GPT-4o 图像生成** | 自然语言多轮编辑 | 45-60% | $0.05-0.10 | 1周 | ⭐⭐ |

### 推荐方案：D（混合流水线）

```
用户原图 + 微调描述
        │
        ▼
  ┌─────────────────────┐
  │ 难度路由器            │
  │ L1→引擎A, L2→引擎B   │
  └─────────┬───────────┘
            │
     ┌──────┴──────┐
     ▼             ▼
  引擎A          引擎B
  Seedream       FLUX
  Inpainting     Kontext
     │             │
     └──────┬──────┘
            ▼
  ┌─────────────────────┐
  │ 质检网关              │
  │ 体型相似度+肤色偏差    │
  │ +面部特征+背景一致     │
  └─────────┬───────────┘
            │
     ┌──────┴──────┐
     ▼             ▼
  通过           不通过
  输出效果图      降级为静态指南
```

**为什么混合**：
- Seedream 身份保持强，适合需要严格保真的 L1 编辑
- FLUX Kontext 指令理解好，适合配饰类编辑
- 双引擎增加总成功率
- 质检网关确保不达标的图不会展示给用户

### 成本优化路径

1. **效果图按需生成**：不是每次都出图。如果 AI 建议是「把袖子放下来」，效果图价值低，不出图只给文字
2. **缓存复用**：相同技巧+相同品类组合的效果图可复用静态示范图
3. **批量折扣**：Seedream 企业批量 $30/1000 张（约 ¥0.21/张）
4. **模型蒸馏**：长期可训练专用小模型降低推理成本
5. **出图率策略**：L1(70%场景) → L2(20%) → L1+L2 出图率 90%，L3(10%) 降级静态图

## 七、产品路线建议

### 关键调整：效果图定位

产品文档 v0.7 中效果图是「核心渲染层」，目标 95% 出图率。基于技术现实，建议调整为：

```
原定位：效果图 = 核心渲染层，必须出图
新定位：效果图 = 体验放大器，能出就出，不能出也够好

具体调整：
  ✅ 效果图出图时：BEFORE/AFTER 对比 + 文字建议
  ✅ 效果图不出时：静态分步图解（手绘/照片） + 文字建议
  ✅ 两种体验都完整可用，效果图是锦上添花
```

**为什么要调**：把产品的核心价值绑定在一个当前技术做不到 95% 可靠性的模块上，是产品最大的系统性风险。核心价值应该是「AI 告诉你今天怎么微调更好看」，这句话可以是效果图说的，也可以是文字+图解说的。

### 调整后的 MVP 范围

| 优先级 | 功能 | 说明 |
|--------|------|------|
| P0 | VLM 穿搭解析 + LLM 建议 | 核心价值交付，不含效果图 |
| P0 | 静态分步图解 | 每个技巧 3 张步骤照片/手绘图 |
| P0 | 打卡系统 + 三元反馈 | 习惯养成引擎 |
| P0 | 身形分析 | 首次设置 |
| P1 | 效果图生成 | 从 P0 降为 P1，作为「实验功能」上线 |
| P1 | 分享功能 | 分享文字+静态对比图（非 AI 效果图） |

### 产品口号微调

```
原来：「AI 模拟效果」+ AI 效果图
调整为：「AI 审稿 + 手把手图解」+（效果图作为实验功能）
```

核心价值主张从「看 AI 把你变好看」转为「AI 教你怎么变好看」——这其实更契合产品的设计哲学：「技巧可携带」。

### 核心假设验证优先级

在投入完整 Phase 1 之前，建议先用 2 周做一个**技术原型验证**：

```
原型目标：验证效果图生成的可行性
原型内容：
  1. 拍 20 组真实穿搭照片（不同光线/角度/品类）
  2. 用 Seedream 4.0 + FLUX Kontext 分别尝试 L1 效果图
  3. 统计首次成功率、质检通过率、人工评价
  4. 输出：「效果图可行性报告」

成本：2周 + ¥500 API费用
决策点：如果 L1 成功率 > 70% → 继续 Phase 1a+b
        如果 L1 成功率 < 50% → 调整产品策略，效果图降级为非核心
```

### ROI 判断

| 场景 | 条件 | 判断 |
|------|------|------|
| 最乐观 | 效果图 80%+ 成功率 | 竞争力极强，值得全力投入 |
| 基准 | 效果图 60-70% 成功率 | 产品可行，效果图作为亮点功能 |
| 保守 | 效果图 < 50% 成功率 | 产品可行但需调整定位为「AI 穿搭教练」而非「AI 效果图」 |
| 最差 | VLM 穿搭解析也 < 70% | 核心循环破裂，需要重新设计 |

**关键结论**：即使效果图完全不可行，产品的核心价值（AI 审稿 + 文字建议 + 静态图解 + 打卡习惯养成）仍然是成立的。效果图是放大器，不是地基。

## 九、总体结论

### 直接回答用户的四个问题

**① 目前的技术大模型能不能完美实现我们想要的效果？**

**不能完美，但能做出可用产品。** 具体来说：
- VLM 穿搭解析 + LLM 建议生成 → 90%+ 可用，几乎完美
- 效果图生成 → 60-75% 可用，不完美但能做
- 身形分析 + 天气 → 95%+ 可用

最大差距在效果图生成。它的技术还在「从研究到产品」的过渡期。12-18 个月后可能成熟。

**② 用什么样的产品路线来设计？**

建议分两步走：
1. **先跑两周技术原型**，用 20 组真实照片验证效果图成功率
2. **根据结果决定**：
   - 成功率 > 70%：效果图作为核心功能，重点投入
   - 成功率 50-70%：效果图作为亮点功能，文字+图解作为主力
   - 成功率 < 50%：调整产品定位，效果图降级，强化文字教练体验

无论如何，**Phase 1 MVP 先不做效果图也能上线验证核心循环**。

**③ 选用什么样的大模型？**

| 用途 | 首选 | 原因 |
|------|------|------|
| VLM 穿搭解析 | 豆包 1.8 Vision | 中文最佳+最便宜（¥0.001/次） |
| LLM 建议生成 | 豆包 1.8 / DeepSeek V3 | 中文质量好+成本可控 |
| L1 效果图 | Seedream 4.0 Inpainting | 身份保持最强+国内部署 |
| L2 效果图 | FLUX.1 Kontext | 指令式编辑最自然 |
| 身形分析 | MediaPipe + VLM 融合 | 免费+够用 |

**④ 投入产出比等关键性内容？**

**投入**：Phase 1 MVP 约 ¥9-17 万（9-13 周，2-3 人），即可上线核心体验闭环。

**风险**：效果图是唯一的技术不确定性。但它是放大器不是地基——产品核心价值不依赖它。

**建议**：先投 ¥500 + 2 周做技术原型验证，再决定是否全量投入 Phase 1。这是最理性的路径。
