# 产品思考

> 创建于 2026-06-29 · 讨论者：kun + Claude

---

## 问题的起源

穿搭推荐项目的核心功能已基本完成。但我意识到一个非常严重的问题：**衣服上传这一步骤会打消大部分用户的兴趣**。

### 当前两种录入方式

| 方式 | 问题 |
|------|------|
| 单件上传（干净背景拍单品） | 太慢，女性用户 100+ 件衣服，逐件上传 5-10 小时 |
| 拍上身照 AI 识别 | 不稳定，无法识别剪裁、面料、背面细节，导致推荐失误 |

### 当前录入流程的技术细节

- 前端：拍照/相册 → Canvas 压缩到 720px/≤500KB（Tailscale Funnel 限制）→ POST /api/wardrobe/add → 轮询 /api/task/{tid} → 审核界面 → 确认入库
- 后端：皮肤检测 → SegFormer 单品分割 → Seedream 白底重构 → 豆包 VLM 分析 → 品类 fuzzy 匹配 → ID 分配 → EXIF 校正 → AI 方向检测 → rembg 抠图 → 缩略图生成 → 标签 JSON → 服装档案.md 更新
- 平均每件耗时 **3-4 分钟**，100 件 = 5-7 小时

---

## 核心洞察

**如果录入是瓶颈，就不要让用户录入。而不是想办法让录入更简单。**

就像 iPhone 用触摸屏取代了键盘——不是让键盘更好用，而是消灭了键盘。

## 产品方向讨论

我们讨论了三个方向：

### 方向 A：衣橱数字化路线（解决录入问题）

**核心命题**：用户有衣服，AI 帮忙搭。用技术创新把录入做到极致简单。

**推荐方案：三层漏斗**
```
第一层（秒级）：电商订单导入 + 品牌数据库匹配（覆盖 30-50%）
第二层（分钟级）：衣柜视频扫描，一次性批量处理（覆盖剩余 50-70%）
第三层（持续）：拍照补录 + 购物时自动添加（零摩擦）
```

**优势**：已有资产全部复用（99 风格 + 评分引擎 + 生图管线 + 抠图处理），护城河深。
**劣势**：即使做到极致仍需 5-10 分钟初始设置；中国市场电商 API 封闭；用户基数天花板有限；仍然是"维生素"产品（nice to have）。
**变现**：订阅制 $5-10/月。
**TAM**：全球 5000 万 - 1 亿 DAU 潜力。

### 方向 B：风格引擎路线（无需衣橱数据）

**核心命题**：用户不需要告诉 AI 自己有什么，AI 告诉用户应该穿成什么样。

**产品形态**：用户输入身形 + 风格偏好 → AI 推荐穿搭方案 + 效果图 + 购买链接 → 用户去买或在自己衣柜里找类似的。

**优势**：零上载摩擦（30 秒完成设置）；直接变现（电商分佣）；病毒传播（效果图天然适合社交分享）；市场规模巨大（中国 3 亿网购服装用户）。
**劣势**：放弃了"真实衣服"的独特价值；与小红书/抖音博主差异化不足；电商分佣在中国利润率极低；效果图理想化 → 退货率高 → 信任崩塌；竞争激烈（大平台都在做）。
**变现**：电商分佣（薄利多销）+ 品牌广告。
**TAM**：中国 3 亿网购服装用户。

### 方向 C：混合路线（推荐方案）

**核心命题**：用方向 B 的低门槛获取用户，用方向 A 的深度功能留住用户并变现。

**用户旅程**：
- Day 1（2 分钟）：输入身形 + 选 3 种风格 → 立即获得 AI 穿搭推荐 + 效果图 + 购买链接
- Week 1：每日推送天气穿搭；"我有类似的"→ 拍照识别 → 自动建虚拟衣橱；"我想买"→ 一键购买；评分反馈 → AI 学习
- Month 1：已不知不觉录入 30+ 件 → 开启衣橱模式 → 精准搭配 → 衣橱分析 → 付费转化

**产品矩阵**：

| 层级 | 功能 | 数据需求 | 商业模式 |
|------|------|---------|---------|
| 🆓 免费层 | 每日 AI 穿搭推荐 + 效果图 + 购买链接 | 身形 + 风格偏好 | 电商分佣 |
| 🥈 基础会员 | 衣橱分析 + 真实衣服搭配 + 利用率追踪 | + 衣橱数据 | 订阅 $5/月 |
| 🥇 高级会员 | 私人 AI 造型师 + 衣橱规划 + 趋势报告 | + 深度偏好 | 订阅 $15/月 |
| 👑 品牌合作 | 趋势数据 + 用户风格画像 + 精准推荐 | 脱敏聚合数据 | B2B SaaS |

**为什么这是最优路线**：
1. 解决了"鸡生蛋蛋生鸡"——先产生价值再获取数据，而非反过来
2. 符合行为心理学——用户是因为"想知道明天穿什么"而用 App，不是"应该管理衣橱"
3. 市场时机正确——2026 年 AI 生成内容接受度空前高，但还没有产品把这个闭环做好

**关键调整**：
- 衣橱数据从"前提条件"降级为"增值功能"
- "被动录入"取代"主动录入"（用户说"我有类似的"→ 自动入库）
- 从"管理过去"转向"设计未来"

## 市场数据

### 核心数据
- ~80% 的衣服常年不穿（ThredUp 2024）
- 大部分用户第一周就放弃衣橱 App
- 一款 15 万用户、运营 5 年的衣橱 App 最终关闭
- 女性平均拥有 100-150 件衣服

### 衣橱 App 五大死因
1. 🧱 **数据录入墙** — 没体验到价值就在设置阶段放弃
2. 🐌 **价值复利太慢** — 录入 10 件几乎无法给出有意义的推荐
3. 🔄 **行为改变太难** — 用户不会突然开始用 App 规划穿搭
4. 💸 **付费意愿极低** — 衣橱管理是"维生素"不是"止痛药"
5. 📉 **衣橱更新滞后** — 买新不录、删旧不删，数据越来越脏

### 全球在线时尚市场
- 全球：**$1.2 万亿**（2026）
- 中国：**$4000 亿**
- AI in Fashion 市场预计 2026-2030 年 CAGR 40%+

## 第二轮思考：放弃 A/B/C，重新出发

> 2026-06-29 续

### 为什么三个方向都不满意

kun 对三个方向的反馈：

| 方向 | 问题 |
|------|------|
| A | 难以解决的衣橱上传难题 |
| B | 同质化内容太多，大厂已在做，发展空间小 |
| C | 融合后产品太不聚焦，没亮点，难吸引用户 |

### 重新定义问题

之前一直在问："**如何让用户把衣服录入系统？**"

真正应该问的是："**用户每天早上最焦虑的是什么？**" —— **"我今天穿什么？"**

这不是衣橱管理问题、不是风格教育问题、不是购物推荐问题。这是一个**每日决策问题**。

最好的日用决策产品参考：
- 🌤️ 天气 App：打开 → 看到天气 → 关掉。零设置。
- 🗺️ 地图 App：输入目的地 → 路线 → 出发。不需要提前录入所有路。
- 🦉 多邻国：打开 → 完成今日任务 → 关掉。连胜打卡。

它们的共同特点：**每天 30 秒解决一个问题，越用越懂你，不需要"设置"。**

## 第二轮方案探索

### 方案一：📸「今日穿搭日记」— 拍照即输入

每天出门前拍一张穿搭，AI 分析 + 打分 + 给一条改进建议。越拍越懂你。

```
Day 1:  拍照 → AI: "不错！白T+牛仔裤是万能公式。
        明天试试把白T换成条纹衬衫，会更有层次感。"
        
Day 30: 拍照 → AI: "你的风格是【Clean Fit 简约休闲】，
        和东京的 xxx 博主风格相似。
        明天 28°C 有雨，建议：浅灰Polo + 深蓝九分裤 + 白色帆布鞋"
```

**用户操作**：打开 → 拍照 → 看到反馈。每天 15 秒。

**kun 的反馈**：方向类似想要的样子，但不能成为真正的产品长时间留住用户——用户完全可以直接打开 AI 对话应用来解决，没必要单独做一款产品。

**教训**：如果产品核心只是"拍照→AI点评"，ChatGPT/豆包/Claude App 随时可以替代。必须做通用 AI 做不到的事。

### 方案 A（新版）：🔥「风格克隆」— 看见喜欢的，秒变自己的

刷小红书/抖音看到喜欢的穿搭 → 截图 → AI 在用户身形上重现这个风格 → 告诉用户怎么穿。

**kun 的反馈**：还是需要有用户的数据，建立衣橱才能实现效果，还是难以解决的老问题。

### 方案 B（新版）：🎮「穿搭挑战」— 每天一个创意约束

Wordle 式的每日穿搭挑战。AI 每天给一个创意限制（如"禁止黑白灰""单色调穿搭"），用户在限制下穿搭，拍照打卡。

**kun 的反馈**：太接近小红书模式了，没有突破点。为什么不直接在小红书上做挑战呢？

---

### 方案 C（新版）：🪞「AR 试穿镜」— 站在镜子前，看到明天的自己

对着镜子拍视频，AI 在身形上叠加明天的穿搭效果。

**kun 的反馈**：目前技术达不到，没法实现。

---

### 方案 D：💡「每天一套」— 穿搭版 Tinder

每天推一套穿搭效果图，用户只需左滑右滑。滑了 30 天，AI 比用户还懂用户适合穿什么。

**kun 的反馈**：产品思路有趣，用陌生人交友的形式探索穿搭风格体验新奇，但没落到真实穿搭需求上，纯粹为了好玩。

---

### 方案 E：💡「风格补给站」— 每天一个微行动

每天给一个具体的、可执行的穿搭微动作。像健身教练一样——不是看冠军身材，而是"今天做 10 个俯卧撑"。

**kun 的反馈**：核心逻辑很认可。市面上所有穿搭应用都在教用户穿什么衣服，而不是教用户如何把衣服穿好。这很重要。

## 核心洞察：时尚最后一公里

### 关键转折点

kun 提出：

> "市面上的所有穿搭应用都是在教你穿什么衣服，而不是教你如何把衣服穿好。这就像是有一个经验丰富的穿搭师，看到你今天的穿搭，不是直接否定这套穿搭，二是进行改进。就如同同样一条丝巾，可以系在脖子里、绑到胳膊上、系到腰带上，这一点点的改变可能就是决定时尚的关键，这些才属于普通用户的时尚盲区。"

> "来选择我们产品的用户肯定都懂基本穿搭，但是离时尚还有差距，我们要解决的就是这时尚最后一公里的问题。"

### 核心洞察重述

```
普通穿搭 App：  "你应该穿这件白衬衫 + 那条黑裤子"  ← 教用户买什么
我们的产品：     "白衬衫解开两颗扣 + 袖口卷两圈"    ← 教用户怎么穿
```

用户不需要被告知"买什么"。用户家里有衣服。用户需要的是——**同一个东西，怎么穿出不同的感觉**。

## 第三轮：穿搭技巧教练方向

### 三个产品概念

#### 概念一：「微调」— 拍一张，改一处

用户拍今天的穿搭 → AI 分析 → 给一个具体的微调（如"衬衫下摆只塞前面一半"）→ 生成改进效果图 → 用户执行后拍照对比 → 打卡

#### 概念二：「点睛」— 穿搭界每日一练

把穿搭技巧拆解成可执行的每日任务，像健身教练一样逐个突破。技巧分四个阶段：基础篇（衣服怎么穿）→ 进阶篇（比例怎么调）→ 高级篇（风格怎么塑）→ 大师篇（自己怎么创）

#### 概念三：「穿对」— 同一套衣服，两种穿法

用户拍照 → AI 生成"同样衣服但穿法不同"的对比图 → 用户直观感受技巧的效果差异

**结论**：三个概念不是互斥的，是一个产品的不同切面。

## 产品逻辑定调：三层递进

kun 提出：

> "真正的产品逻辑应该是，指导用户怎么穿好看，搜集到用户足够的数据后开始推荐用户穿什么，再之后才是指导用户该怎么买，层层递进，逐步提高用户使用率和留存率。"

> "指导用户怎么穿好看，采用用户拍照，AI分析思考给出建议，并给出改进后的效果图，供用户参考去做。后续的AI推荐穿搭有足够多的用户数据后，可以做成类似于穿搭版 Tinder，这时候也有了用户的服装数据，可以进一步去推荐购买了。"

> "最终目的是要做一个轻量化但是能长时间留住用户的应用，让用户使用起来无负担。"

### 三层递进架构

```
Layer 1：学会穿
拍照 → AI微调 → 前后对比 → 打卡
用户获得：穿搭技巧 + 每天肉眼可见的进步
产品获得：身形数据 + 衣橱数据 + 风格偏好 + 行为数据
周期：Day 1 ~ Day 30

        ↓ 数据够了，自然过渡

Layer 2：帮你穿
AI知道用户有什么 + 喜欢什么 → 每天推送方案
左滑右滑 → AI越来越懂用户
周期：Day 30+

        ↓ 衣橱数字化完成 + 风格DNA形成

Layer 3：聪明买
AI发现衣橱缺口 → 精准推荐单品
不是"猜你喜欢"，而是"你缺这件来完成你的风格"
周期：Day 60+
```

### 产品定位一句话

> 每天 1 个穿搭微调。不教你买什么，教你**怎么穿**。

### 为什么这个产品 AI 对话 App 做不到

| | 我们 | ChatGPT |
|---|---|---|
| 持续追踪你的穿搭 | ✅ 每张照片都在积累 | ❌ 每次对话从零开始 |
| 在你身形上生效果图 | ✅ 专用模型 | ❌ 没这个能力 |
| 结构化技巧课程 | ✅ 100+技巧 × 分级 | ❌ 碎片化建议 |
| 知道你的衣橱 | ✅ 被动积累 | ❌ 不知道 |
| 知道你的进步轨迹 | ✅ 历史对比 | ❌ 只有当前对话 |
| 习惯养成引擎 | ✅ 打卡/连胜/成就 | ❌ 没有 |

## 第四轮：产品设计文档批改讨论

> 2026-06-29 · 讨论者：kun + Claude

### 讨论前提

产品核心卖点方向正确：帮助用户改进穿搭而非决策，在尊重用户喜好和实际情况的基础上，用 AI 分析出合理可操作的提升方向。

以下三个问题需要修正设计文档。

---

### 问题一：被动衣橱数字化 — 照片提取服装不可行

**kun 的反馈**：前一个产品已经测试过从真实上身照片中提取服装平面图（白底重构），可操作性很低。分割不稳定、面料难判断、版型不准确。需要更清晰简单且确保准确度的方案来记录服装信息。

**分析**：之前的设计（产品设计文档 2.2）期望通过每日拍照自动提取单品、建立虚拟衣橱。这个路线技术上不成熟——SegFormer 分割 + 白底重构的准确度远不足以支撑稳定的衣橱数据。

**解决方案：放弃单品级衣橱，改用「穿搭快照」**

核心思路：不把衣服从照片里"切出来"，而是用 AI 描述这张照片里有什么、怎么穿的。

```
之前的设计（不可行）：
  用户拍照 → 分割单品 → 白底重构 → 单品入库 → 虚拟衣橱
  ❌ 分割不准、面料误判、版型难判断

改进后（可行）：
  用户拍照 → AI 直接输出结构化穿搭描述 JSON
  {
    "items": [
      {"category": "TS", "color": "白色", "fit": "宽松", "note": "圆领短袖"},
      {"category": "PT_denim", "color": "深蓝", "fit": "直筒"}
    ],
    "styling_notes": "T恤自然垂放，裤脚无卷边"
  }
  ✅ 品类+颜色 AI 识别 >90%准确，版型能做到七八成，够用
```

**关键改变**：
- 衣橱从「单品库」变成「穿搭日历」——用户看到自己的穿搭历史，不是衣橱货架
- Layer 2 推荐时，AI 不说"穿你的第 17 号白 T 恤"，而是说"你 6 月 29 日穿的那件白 T 恤 + 这次试试深灰西裤"
- 不需要精确到品牌型号，品类+颜色+版型就足够做搭配推荐了

**对产品设计文档的影响**：
- 2.2「被动衣橱数字化」表述需修正
- 6.1 用户数据模型中「单品识别结果」改为「穿搭快照 JSON」
- 4.2 Layer 2 体验中「你衣柜里都有 ✅」需改为基于穿搭历史的匹配逻辑
- 项目资产索引中 `extract_clothing.py`（白底重构）从 Phase 1 降为可抛弃

---

### 问题二：穿搭小白的起点 — 错误基础上微调 = 错上加错

**kun 的反馈**：如果是穿搭小白，拍出来的穿搭一上来就是错误的，在此基础上再怎么优化也是错的更多。

**分析**：当前设计假设所有用户都"会基本穿搭"。但实际上零门槛产品必然会有小白用户。如果基础分只有 20 分（比如严重不合身的裤子），微调裤脚卷边毫无意义。

**解决方案：AI 先评分再定策略 —— 三级分流**

```
🔴 基础分 < 40（存在严重问题）
   策略：不给微调，给 1 个关键纠正
   示例："这条裤子对你来说太长了，建议先换成九分裤或拿去改短。
         调整后再拍照，我来教你下一步。"
   本质：告诉用户"问题出在哪"，而不是"怎么微调"

🟡 基础分 40-65（方向对，细节差）
   策略：正常微调模式
   示例："把上衣塞进去，腰线立刻清晰。"

🟢 基础分 65+（不错，精进）
   策略：给进阶技巧
   示例："今天试试加一条与鞋同色的腰带。"
```

**另一个信号源**：首次身形拍照时，用户穿什么来拍照本身就是穿搭水平的信号。AI 可以在身形报告中给出一个「穿搭起点评估」，让用户知道自己在哪个阶段。

**对产品设计文档的影响**：
- 3.2 AI 审稿引擎需要增加「基础分评估」步骤（在「解析今日穿搭」之后）
- 3.4 每日交互流程需要增加 🔴 纠正模式的展示
- MVP 需要支持三个级别的差异化输出

---

### 问题三：动画演示成本高 + 纯学习收获感低

**kun 的反馈**：
1. 动画演示过程成本太高，目前无法实现
2. 在没有真实穿搭场景的情况下纯粹学习技巧，用户收获感不高

**分析**：
- 子问题 1（动画）：3.3 技巧知识库中的 `visual_guide_url` 和 3.4 场景 B 中的「动画演示：塞衣过程」在当前阶段不可行。每个技巧一个动画，制作成本极高。
- 子问题 2（纯学习）：场景 B「今日技巧卡」——用户不拍照只看技巧——缺少真实穿搭场景锚定，看了就忘，跟刷短视频没有区别。收获感低，留存差。

**解决方案 A：动画替换为静态分步图解 + 用户自己的 BEFORE/AFTER**

```
动画 → 替换为：
  ✅ 静态分步图解（3-4 张真人示范照 + 文字标注，一次拍摄永久使用）
  ✅ 用户自己的历史 BEFORE/AFTER（Day 7 回看 Day 1，变化肉眼可见）
  ✅ 原项目已有的小红书/Instagram 采集穿搭图作为技巧示意
```

**解决方案 B：取消独立的「不拍照模式」，改为拍照后的延伸**

```
之前的设计：
  主路径：拍照 → 微调
  副路径：不拍照 → 技巧卡（独立学习）

改进后：
  唯一主路径：拍照 → AI 审稿 → 微调建议 + 技巧原理
  拍照是每天的核心动作，跟多邻国一样——不打开就没有今日进度
  
  如果用户真的不想拍照：
  → 推送"回顾你过去 7 天的进步"（自己的照片最有动力）
  → 推送"明天天气，适合尝试 XX 技巧"（预告而非学习）
```

**核心原则**：技巧学习嵌入实践，用户在做中学，而不是先学再做。技巧树的解锁条件是"你用过这个技巧"而非"你看过这个技巧"。

**对产品设计文档的影响**：
- 3.4 场景 B「今日技巧卡」需要删除或大幅弱化
- 3.3 技巧数据结构中 `visual_guide_url` 改为 `visual_guide_images`（静态图数组）
- 3.6 游戏化系统中「技巧解锁条件」从"看过"改为"用过"
- 3.1 体验循环中去掉独立学习路径

---

### 本轮总结：需要修正的设计点

| # | 原设计 | 问题 | 修正方向 |
|---|--------|------|---------|
| 1 | 被动衣橱数字化（单品提取入库） | 技术不可行 | 穿搭快照 JSON（品类+颜色+版型标签） |
| 2 | 对所有人统一的微调策略 | 小白起点问题 | 三级分流：纠正 / 微调 / 精进 |
| 3 | 动画演示 + 独立技巧卡模式 | 成本高 + 收获感低 | 静态图解 + 取消独立学习，嵌入实践 |

## 第五轮：Karpathy 视角审查 + 季节衰减讨论

> 2026-06-29 · 讨论者：kun + Claude（Karpathy 视角）

### Karpathy 审查发现的 6 个问题

1. **8 步 AI 管线可靠性连乘**：每步 90% → 端到端 43%。解决方案：拆成 3 段式独立架构（穿搭描述→建议生成→效果图），每段独立降级
2. **VLM 评分不可靠**：连续分值校准极差。解决方案：用 7 项 Yes/No checklist 代替 0-100 打分，分流规则基于 ❌ 数量
3. **身形分析冷启动**：3 张照片置信度不足。解决方案：展示置信度 + 允许手动修正 + 每张新照片持续校准
4. **效果图不忠于原图**：Seedream 会改体型/肤色/衣服。解决方案：L1-L4 难度分级 + L1 用 inpainting 保持保真度 + 失败自动降级为静态指南
5. **穿搭快照无时效**：去年的白T恤衰减到零。解决方案：季节标签 + 季节修正权重（见下方深入讨论）
6. **缺少效果反馈闭环**：AI 不知道建议是否真的有用。解决方案：打卡时三元反馈（更好/差不多/不如）

### 季节衰减深入讨论

**kun 的疑问**：如果季节更替了，或者过了一年的夏天，要怎么记录这些信息？

**分析**：纯时间衰减无法处理季节循环——去年夏天的白T恤，今年夏天应该重新活跃，但因为过了一年权重衰减到零。

**解决方案**：每张穿搭快照在拍照时通过天气 API 自动标记季节和温度标签，用户无感知。检索时用「基础时间衰减 + 季节修正」：
- 同季节快照（去年夏天 → 今年夏天）：季节修正 +0.5，从 0.15 提升到 0.65
- 反季节快照（冬天 → 夏天）：季节修正 -0.3，最低 0，不推荐
- 换季月份（3月/9月）：相邻季节修正为 0，两季权重接近，自然过渡

季节-月份映射：12-2月冬 / 3-4月春 / 5-6月初夏 / 7-8月盛夏 / 9-10月秋 / 11月深秋

**对产品设计文档的影响**：
- 6.1 穿搭快照增加 season/temp_range/weather 字段
- 新增 6.3 穿搭快照季节系统章节
- 8.2 Phase 2 增加「季节感知检索」

## 第六轮：男女区分 + 城市/工作/季节策略

> 2026-06-29 · 讨论者：kun + Claude

### 核心问题

kun 提出：穿搭应用是否应该区分男女用户？城市、工作、季节需不需要做规定？所有内容做到一套程序还是引导用户进入不同程序？

### 竞品调研关键发现

- 没有任何主流竞品做男女分离的独立 App
- Style DNA（300万用户）、OneOff、Outfit.ly、Alta 等全部是统一 App 内做性别选择
- OneOff 还额外提供 "genderless" 选项
- 天气集成是所有主流竞品的标配能力

### 决策：架构预留，内容聚焦

**一句话**：架构从 Day 1 支持多性别（`target_gender` 字段），内容 Phase 1 只做男性。

**为什么架构要预留**：每条内容加一个 `target_gender` 字段的成本不到半天。但如果现在不加，未来每条内容都要逐条回溯、改 prompt、改数据结构，成本翻 5 倍以上。

**为什么内容先做男性**：
- 原项目已有 49 个男装风格 + 18 个男性评分指纹，内容资产最厚
- 身形分析 + 效果图生成先调好一套，再扩展到女性
- 种子用户聚焦一个群体

**技巧库性别分布**：通用 ~40%、男性专属 ~30%、女性专属 ~30%

### 其他维度决策

| 维度 | 决策 | 原因 |
|------|------|------|
| 季节 | Phase 1 就做，天气 API 自动标记 | 实现成本极低，影响推荐准确度 |
| 城市 | 不做独立维度，= 天气坐标 | 穿搭需求由温度+天气决定，城市文化差异由风格偏好覆盖 |
| 工作场景 | Phase 2 可选标签 | 对 MVP 不是必要的，用户风格偏好已隐含场景信息 |
| 儿童 | 不做，独立评估 | 品类体系/身形模型/购买决策完全不同 |

### 对产品设计文档的更新
- 技巧库增加 `target_gender` 字段
- 首次体验增加性别选择
- Phase 1 明确为男性用户
- Phase 2 增加女性用户扩展
- 未决问题中对应项目标记为已解决

## 第七轮：竞品差距分析 + 行业踩坑研究

> 2026-06-29 · 讨论者：kun + Claude

### 核心问题

kun 提出三个问题：
1. 跟市面上已有产品对比，我们还有什么没考虑到的？
2. 这类产品容易踩的坑有哪些？
3. 我们的核心优势大到足以让用户选择我们吗？

### 竞品研究关键发现

**行业五大死因**（按致死率排序）：

1. **技术不稳定导致数据丢失**（致命）：Combyne 更新删用户数据，Essembl 丢失 400+ 件衣橱
2. **先让用户投入再索要付费**（致命）：GetWardrobe、Outfity、Acloset 被骂"骗子"
3. **AI 推荐质量差**（重伤）：Cladwell 夏天推荐毛衣，AI 穿搭千篇一律
4. **没有日常打开的理由**（重伤）：整个衣橱管理品类留存低，录完就弃
5. **虚拟试穿效果失真**（重伤）：Doppl 身体变形，AI 模特六根手指

**我们产品 vs 竞品核心差异**：

| 维度 | 衣橱管理 | AI试穿 | 通用AI | 我们 |
|------|---------|--------|--------|------|
| 初始摩擦 | 极高(2h+) | 中 | 低 | 极低(60s) |
| 日常打开理由 | 无 | 无 | 弱 | 每日拍照打卡 |
| 数据积累 | 手动 | 无 | 无 | 被动积累 |
| 可携带价值 | 无 | 无 | 无 | 技巧离开App也能用 |

### 发现的 5 个盲区及补充

| # | 盲区 | 补充到设计文档 |
|---|------|-------------|
| 1 | 数据安全与隐私 | 6.2 重写：端到端加密、数据导出权、删除权、每日备份、照片不用于训练 |
| 2 | 效果图保真度标准 | 6.2 硬约束：不改体型/肤色/面部/背景，写入 negative prompt |
| 3 | 变现"先给价值再取钱" | 7 重写：免费层完整可用、会员是扩展非解锁、MVP 全免费 |
| 4 | 离线/弱网降级 | 3.2：拍照本地缓存、后台异步分析、超时自动降级 |
| 5 | 个性化防趋同 | 3.2：个性化因子（风格契合度+历史反馈-重复推荐-全局流行度） |

### 差异化评估结论

我们的核心差异化（零录入 + 每日打卡 + 技巧可携带）**足以让用户选择我们**——前提是三个硬骨头啃下来：
1. 效果图保真度 ≥70%
2. 零数据丢失
3. 不搞"先投入后付费"

## 第八轮：Karpathy 第二轮审查 + 用户等级策略

> 2026-06-29 · 讨论者：kun + Claude（Karpathy 视角）

### Karpathy 第二轮审查发现

**三个自相矛盾**：
1. 目标用户「已会基本穿搭」vs 纠正模式大量设计 → 收紧目标用户，纠正模式兜底不宣传
2. 「必须拍照」vs「不想拍照时的体验」→ 可浏览自己数据，不含内容消费
3. 评分数字来源不明 → 补充 Checklist→分值映射（YES占比×100）

**两个不合理**：
4. 3.5 Step 3 残留「看技巧卡」按钮 → 删除
5. 效果图被当附加功能 → 提升为核心渲染层，Phase 1 重点投入

**四个放大优势建议**：
6. 新增 1.5「为什么通用 AI 做不到」— 护城河声明
7. 新增「AI 了解你」指标 — 数据飞轮可见，增加迁移成本
8. 新增⑤技巧内化 — 用户感知到「技巧可携带」
9. 新增分享功能（P0）— BEFORE/AFTER 是天然病毒内容

### 用户等级策略讨论

kun 提出：如何做到让不同水平的用户都适用？需不需要让用户选择等级？

**结论**：不需要。三级分流（🔴纠正/🟡微调/🟢精进）自动判断每日穿搭状态，不是用户能力等级。同一个用户周一纠正、周五精进是正常的。

关键改动：
- Checklist 定位从「能力等级」→「今日穿搭状态」
- UI 隐藏评分数字和颜色标记，改为自然语言
- 大师篇由数据条件触发（15 技巧 + 90 天），不需用户选择

### 对产品设计文档的更新
- 新增 1.5 节 + 体验循环⑤步骤 + AI了解你指标 + 分享功能
- 目标用户收紧、3.4 UI 改为自然语言、3.5 删看技巧卡
- 效果图从段三提升为核心渲染层
- Checklist 分流定位修正 + 评分映射公式
- 大师篇数据触发说明

**Karpathy 核心判断**：

- 三段式独立管线架构正确——系统可靠性取决于尾部行为
- 先做最简单版本。Don't be a hero.
- 模型选择 < 数据积累 < 用户反馈循环

### 项目命名：搭调 (DressTune)

「搭调」= 真实中文词，意为"和谐、匹配、对路"。
- 搭 = 穿搭。衣品，搭配功力。
- 调 = 微调。每天调一点，从路人到博主的那一口气。
- 合起来：让你的穿搭更搭调。
- 英文 DressTune：Dress + Tune，像给乐器调音一样给穿搭微调。

备选：对镜、一调、穿感（均被否决，理由详见开发计划书）。
